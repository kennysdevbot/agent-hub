## System Overview

```

│              Web Dashboard (React)           │
│  Tailwind CSS · Zustand · RTK Query         │
│  @supabase/supabase-js (anon key)           │
└──────────────────┬──────────────────────────┘
                   │ Supabase REST API (PostgREST)
```

No custom backend server. The frontend and agents both talk directly to Supabase. Kenny views everything through the web dashboard.

## API Endpoints (Auto-Generated by Supabase PostgREST)

All CRUD is handled by Supabase's auto-generated REST API. No custom route handlers needed.

```
# Supabase REST pattern: /rest/v1/{table}?{filters}

GET    /rest/v1/projects                          # List all projects
GET    /rest/v1/projects?id=eq.{id}               # Get one project
POST   /rest/v1/projects                          # Create project
PATCH  /rest/v1/projects?id=eq.{id}               # Update project
DELETE /rest/v1/projects?id=eq.{id}               # Delete project

GET    /rest/v1/tasks?project_id=eq.{id}          # Tasks for a project
POST   /rest/v1/tasks                             # Create task
PATCH  /rest/v1/tasks?id=eq.{id}                  # Update task

GET    /rest/v1/agents                            # List agents
PATCH  /rest/v1/agents?id=eq.{id}                 # Update agent status

GET    /rest/v1/activity_logs?order=timestamp.desc # Activity feed
POST   /rest/v1/activity_logs                     # Log activity
```

For complex queries (e.g., dashboard stats), use Supabase RPC with Postgres functions:
```
POST   /rest/v1/rpc/get_dashboard_stats           # Custom Postgres function
```

## Dashboard Views

1. **Overview** — active projects, agent status cards, recent activity feed
2. **Project Detail** — task board (kanban-style), progress stats
3. **Activity Log** — filterable table: by agent, by project, by date range
4. **Agent View** — per-agent workload, history, current task

## Key Decisions

- **Supabase over SQLite:** See full reasoning below. TL;DR: Supabase eliminates the need for a custom Express backend entirely — the frontend and agents talk directly to Supabase's auto-generated REST API. Less infra, less code, faster to ship.
- **Zustand + RTK Query:** Zustand for ephemeral UI state (filters, modals). RTK Query for everything server-side (auto-caching, invalidation). RTK Query hits Supabase REST endpoints directly.
- **Agent API first:** Agents use `@supabase/supabase-js` or raw HTTP to insert activity logs and update task status. No wrapper server needed.

## DB Decision: Why Supabase

### Options Evaluated

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| **SQLite + better-sqlite3** | Zero-config, file-based, fast | Requires self-hosted Express server; agents need a running backend to write; no built-in REST API | ❌ Too much infra for what we need |
| **Supabase** | Hosted Postgres, auto-generated REST API (PostgREST), realtime subscriptions, Row Level Security, JS client, 500MB free | Free tier pauses after 7 days inactivity (easy workaround); 2 free projects max | ✅ **Winner** |
| **Turso (libSQL)** | Edge SQLite, generous free tier (5GB, 500M reads/mo), SQLite-compatible | Still need a custom backend for the frontend; no built-in REST API layer; raw SQL only | ❌ Great DB, but doesn't eliminate backend |
| **Neon (Postgres)** | Hosted Postgres, 0.5GB free, scales to zero | Same as Turso — no REST API layer, need custom backend; cold starts on scale-to-zero | ❌ Good DB, still need Express |
| **PlanetScale** | Was great, but removed free tier in 2024 | No free tier | ❌ Dead for us |
| **Firebase/Firestore** | Easy setup, generous free tier, realtime | NoSQL — bad fit for relational data model (tasks → projects, agents → tasks); vendor lock-in; awkward querying | ❌ Wrong data model |
| **JSON files** | Simplest possible | No queries, no concurrency, won't scale, no API | ❌ Not viable |

### Why Supabase Wins

1. **Eliminates the Express backend entirely.** Supabase gives us a PostgREST API out of the box. The React frontend hits Supabase directly via `@supabase/supabase-js`. Agents do the same — `supabase.from('activity_logs').insert({...})`. No Express, no deployment, no server to maintain.

2. **Relational model fits perfectly.** Our schema (agents, projects, tasks, activity_logs with foreign keys) is textbook relational. Postgres handles this natively. No impedance mismatch.

3. **Agents can write directly.** With a service role key (or scoped RLS policies), agents POST activity logs and update task status via the JS client or raw HTTP. Zero custom API code needed.

4. **Frontend reads directly.** RTK Query can hit Supabase REST endpoints. We get auto-generated endpoints for every table — `GET /rest/v1/tasks?project_id=eq.xxx`, etc. Filtering, sorting, pagination all built in.

5. **Realtime for free.** If we want the dashboard to live-update when agents log activity, Supabase realtime subscriptions handle it with no WebSocket server to build.

6. **Free tier is sufficient.** 500MB is massive for an internal tool tracking a 4-agent team. We'll use maybe 1-5MB over the lifetime of this project. Unlimited API requests.

7. **The auto-pause concern is manageable.** Free tier pauses after 7 days of inactivity. Since agents are writing activity logs regularly during active use, this won't trigger. For idle periods, a simple cron ping (or GitHub Action) keeps it alive. If it does pause — it un-pauses on next request in ~60 seconds.

### What This Changes

- **Remove:** Express backend, `better-sqlite3`, all API route handlers
- **Add:** `@supabase/supabase-js` client in frontend and agent scripts
- **Simplify:** Deployment becomes "deploy a static React app" (Vercel/Netlify/GitHub Pages) — no server
- **Schema:** Same tables, just created in Supabase dashboard or via SQL migration
- **Auth/Security:** Row Level Security (RLS) policies control who reads/writes what. Service role key for agents, anon key for dashboard (read-only via RLS).

### Migration from Original Plan

The data model (tables, columns, types) stays identical. We just:
1. Create the tables in Supabase instead of SQLite
2. Replace Express routes with direct Supabase client calls
3. Set up RLS policies (dashboard = read-only, agents = read/write)
4. Deploy frontend as a static site
